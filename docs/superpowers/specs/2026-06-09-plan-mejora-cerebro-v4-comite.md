## ⚠️ RIESGOS NO ESTIMADOS QUE CAZÓ EL COMITÉ

1. **Denominador real del boot**: el harness inyecta ~30 system-reminders + catálogo de skills + instrucciones MCP ANTES de leer una neurona. El techo de ~9k aplica solo a la fracción que el cerebro controla; optimizar el ancla a 1.200 chars puede ser ruido frente a un overhead 10-20× mayor. → Medir baseline de sesión VACÍA antes de fijar el techo; el apalancamiento real es DIFERIR superficie, no comprimir.
2. **Concurrencia con el cron CI**: el cron (bumpea main cada 4h) es el único writer no-humano; puede pisar el bundle/tag baseline entre la medición y el tag → ventana controlada.
3. **Push→GitHub Pages sin staging**: un KERNEL corrupto commiteado se PUBLICA al push; el enforcement vive en pre-commit (saltable con `--no-verify`). El punto de no-retorno es el push, no el commit → riesgo residual declarado.
4. **Git-Bash en la máquina del cliente**: el pre-commit es `#!/bin/sh`. Si solo hay PowerShell, el hook nunca corre y el "guardián determinista" es decorativo → confirmar o reescribir el hook en node.
5. **Costo de proceso-git en Windows**: cada `execSync` arranca git.exe (~30-80ms); el plan añade ~7 checks → boot lento que compite con la economía que persigue.
6. **Segundo cerebro de metadata**: manifest + baseline + subsumed-map + conflict-log + fixtures + READMEs = superficie nueva que puede desincronizarse; a N=3-4 su costo de gobernanza puede superar al cerebro que ordena.
7. **Fidelidad semántica de Fase B no enforceable**: el linter verifica conservación cuantitativa (hash/IDs), NO que el resumen sea fiel → aprobación humana por pieza no-negociable; la fidelidad va al Comité #2.
8. **Honestidad del verificador a largo plazo**: un fixture stale tras evolucionar el SCHEMA da falso-verde (viola M-10) → la maquinaria de verificación entra a su propio ciclo de frescura.
